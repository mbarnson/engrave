    Create `tests/unit/test_chunker.py` with tests:
    - chunk_score on simple_score.ly splits at the rehearsal mark boundary
    - chunk_score falls back to 4-8 bar chunks when no structural cues exist (use a plain score with only bar checks)
    - chunk_score produces 1-2 bar overlap between adjacent chunks (design choice: overlap captures pickup notes and cadential material continuity)
    - expand_repeats unrolls `\repeat volta 2 { A } \alternative { { B } { C } }` into `A B A C`
    - chunk_score on repeat_score.ly expands repeats before chunking (design choice: unrolling produces linear chunks that match how performers read through the form)
    - chunk_score on multi_part_score.ly produces both single-instrument chunks AND full-score vertical chunks
    - Each chunk contains the LilyPond source fragment and bar range metadata
    - Chunks are not compilable on their own (fragments only, per research pitfall 4)

    **GREEN phase — implement to pass:**

    Create `src/engrave/corpus/chunker.py`:

    1. `BOUNDARY_PATTERNS` — list of compiled regex patterns for structural boundaries (from research):
       - `\mark\s+\default`, `\mark\s+\d+` (rehearsal marks)
       - `\bar\s+"\\|\\|"` (double barlines)
       - `\bar\s+"\\.:` (repeat barlines)
       - `\key\s+\w+\s*\\(major|minor)` (key changes)
       - `\time\s+\d+/\d+` (time signature changes)
       - `\repeat\s+(volta|segno)` (repeat structures)
       - `\section`, `\fine`, `\segnoMark`, `\codaMark`

    2. `find_phrase_boundaries(ly_source: str) -> list[int]` — find character positions of structural boundaries using regex patterns. Return sorted, deduplicated list.

    3. `expand_repeats(ly_source: str) -> str` — expand `\repeat volta N { body } \alternative { { alt1 } { alt2 } }` into unrolled form (body + alt1, body + alt2, etc.). Also handle `\repeat volta N { body }` without alternatives (just repeat N times). Handle D.S./coda by expanding `\segnoMark` and `\codaMark` blocks into linear form. Use regex + bracket counting for block extraction. This is a best-effort expansion — complex nested repeats may not fully unroll, and that's acceptable.

    4. `_count_bars(ly_fragment: str) -> int` — count bar checks (`|`) in a LilyPond fragment to determine bar count. Use python-ly tokenizer if available, fall back to regex counting of `|` outside strings/comments.

    5. `_split_at_boundaries(ly_source: str, boundaries: list[int], target_bars: tuple[int, int] = (4, 8), overlap_bars: int = 1) -> list[tuple[str, int, int]]` — split source at boundary positions, merging adjacent small chunks to hit 4-8 bar target. Each chunk tuple: (source_fragment, bar_start, bar_end). Add overlap_bars (default 1, chosen for pickup notes and cadential material continuity) from the preceding chunk's tail to each chunk's head.

    6. `_extract_instrument_parts(ly_source: str) -> dict[str, str]` — parse a multi-part score to extract individual instrument voices. Use python-ly or regex to find `\new Staff` / `\new Voice` blocks and their content. Return dict of instrument_name -> ly_source.

    7. `chunk_score(ly_source: str, source_path: str, source_collection: str, header_metadata: dict | None = None) -> list[dict]` — main entry point:
       a. Call expand_repeats to unroll repeat structures (design choice: linear chunks match performer reading order)
       b. Call find_phrase_boundaries to find structural breaks
       c. Call _split_at_boundaries with 1-bar overlap for pickup/cadential continuity
       d. For multi-part scores (detected by presence of multiple `\new Staff`): produce BOTH single-instrument chunks (one per instrument per bar range) AND full-score vertical chunks (all instruments at same bar range)
       e. Return list of dicts with keys: source (fragment), bar_start, bar_end, chunk_type ("single_instrument" or "full_score"), instrument (name or "all"), chunk_index
       Note: These dicts are raw chunker output — Plan 03's pipeline will convert them to Chunk model objects with full metadata.

    Use python-ly (`ly.lex`, `ly.document`) for tokenization where it helps (bar counting, instrument extraction). Fall back to regex when python-ly is insufficient. Import python-ly with a try/except to handle potential Python 3.12 compatibility issues gracefully — if python-ly fails to import, use regex-only fallback with a logged warning.

    **REFACTOR** if needed: extract shared regex patterns, ensure consistent bar counting.

    - `extract_metadata(ly_fragment: str, bar_start: int, bar_end: int, header_metadata: dict | None = None) -> dict` — extract all metadata fields from a LilyPond fragment:
      - `key_signature`: regex for `\key <note> \<mode>`, format as "C major", "F# minor", etc.
      - `time_signature`: regex for `\time <num>/<denom>`
