    **TDD: RED phase first, then GREEN.**

    Create `src/engrave/midi/__init__.py` re-exporting public API.

    **loader.py**: Implement MIDI loading and type 0/1 normalization using mido:
    - `NoteEvent` dataclass: pitch (int 0-127), start_tick (int), duration_ticks (int), velocity (int 0-127), channel (int 0-15)
    - `MidiTrackInfo` dataclass: track_index, channel (int|None), program (int|None), instrument_name (str|None), notes (list[NoteEvent]), is_drum (bool)
    - `load_midi(path: str) -> tuple[list[MidiTrackInfo], dict]`: Load file, return normalized tracks + global metadata dict (type, ticks_per_beat, num_tracks)
    - Type 0: split single track by MIDI channel. Track note_on (velocity>0) and note_off (or note_on velocity=0) to build NoteEvent pairs. Channel 9 (0-indexed) is drums. Extract program_change per channel for instrument names.
    - Type 1: parse each track separately, extract instrument name from track_name meta messages and program_change messages. Skip tracks with no note events (conductor/tempo tracks).
    - `_gm_instrument_name(program: int) -> str`: Lookup General MIDI instrument name from program number. Use a simple dict of the 128 GM instrument names.
    - Raise ValueError for unsupported MIDI types (not 0 or 1).

    **analyzer.py**: Implement musical analysis using pretty_midi:
    - `MidiAnalysis` dataclass: key_signature (str, e.g. "c \\major"), time_signatures (list of (numerator, denominator, tick) tuples), tempo_changes (list of (bpm, tick) tuples), instruments (list of instrument name strings), total_bars (int), ticks_per_beat (int)
    - `analyze_midi(path: str) -> MidiAnalysis`: Use pretty_midi.PrettyMIDI to extract key estimation (via get_chroma then map to key name), tempo changes (get_tempo_changes), time signature changes, instrument list. Estimate total bars from total duration and time signature.
    - Key estimation: compute chroma vector, find dominant pitch class, use Krumhansl-Kessler profiles (or pretty_midi's built-in approach) to determine major/minor. Format as LilyPond key string.

    **Test fixtures**: Create 3 small programmatic MIDI files using mido in test setup (conftest or inline):
    - `simple_type0.mid`: Type 0, 2 channels (piano + bass), 4 bars of simple notes, program_change messages
    - `simple_type1.mid`: Type 1, 2 tracks with track names and program_change, 4 bars
    - `no_metadata.mid`: Type 1, 2 tracks with NO track names, NO program_change -- tests the "no metadata" warning path

    Generate MIDI fixtures programmatically in a `tests/fixtures/generate_midi_fixtures.py` helper script, or inline in conftest. The fixtures should be deterministic and small.

    **TDD: RED phase first, then GREEN.**

    **tokenizer.py**: Convert MIDI note events to human-readable text for LLM prompts:
    - `tokenize_section_for_prompt(notes: list[NoteEvent], time_sig: tuple[int,int], key: str, bars: tuple[int,int], ticks_per_beat: int) -> str`
    - Output format per bar: `bar N: pitch(duration, velocity) pitch(duration) ...`
    - Pitch naming: MIDI number to LilyPond note name with octave. Use note names (c, cis, d, dis, e, f, fis, g, gis, a, ais, b) -- sharps only for simplicity, no enharmonic context yet. Octave marks: c (MIDI 48-59 region) = no mark, c' (60-71) = one tick, c'' (72-83) = two ticks, etc. Follow LilyPond absolute pitch convention.
    - Duration quantization: snap note durations to nearest standard LilyPond duration (1=whole, 2=half, 4=quarter, 8=eighth, 16=sixteenth). Use ticks_per_beat for grid calculation. Handle dotted notes (duration * 1.5).
    - Velocity to dynamic: only include when velocity changes from previous note. Map velocity ranges to pp/p/mp/mf/f/ff.
    - Include rests explicitly where there are gaps in the note stream.
    - Group by bar number. Bar number derived from tick position and time signature.

    **sections.py**: Detect section boundaries in MIDI:
    - `SectionBoundary` dataclass: bar_number (int), boundary_type (str: "rehearsal_mark"|"time_sig_change"|"key_sig_change"|"tempo_change"|"fixed_length"), tick (int)
    - `detect_sections(midi_path: str, min_bars: int = 4, max_bars: int = 8) -> list[SectionBoundary]`
    - Priority chain for boundary detection:
      1. Rehearsal marks from MIDI meta events (marker messages)
      2. Time signature changes
      3. Key signature changes
      4. Significant tempo changes (>10% change)
      5. Fixed-length fallback (every max_bars bars)
    - Use mido to scan for meta events. Convert tick positions to bar numbers using time signature.
    - Sections must be between min_bars and max_bars. If a structural boundary creates a section shorter than min_bars, merge with previous section.
    - Return list of SectionBoundary objects marking where each section starts.

    **Unit tests** (test_midi_tokenizer.py):
    - test_tokenize_single_note: one note produces correct pitch name, duration, velocity
    - test_tokenize_pitch_naming: MIDI numbers map to correct LilyPond note names (c, c', c'', etc.)
    - test_tokenize_duration_quantization: various tick durations snap to correct LilyPond durations
    - test_tokenize_dotted_notes: dotted durations detected correctly
    - test_tokenize_velocity_changes_only: velocity only appears when it changes
    - test_tokenize_rests_included: gaps produce explicit rest tokens
    - test_tokenize_multiple_bars: output grouped by bar number with correct format
    - test_tokenize_bar_numbers_correct: bar numbers match tick positions for given time sig

    **Unit tests** (test_midi_sections.py):

    - test_detect_merges_short_sections: boundary creating <min_bars section merges with previous
    - test_detect_respects_max_bars: no section longer than max_bars
    - test_detect_priority_chain: multiple boundary types present, higher priority wins
