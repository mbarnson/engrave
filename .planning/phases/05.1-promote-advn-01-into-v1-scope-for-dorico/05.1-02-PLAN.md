   - Returns the suffix appended to the shared prompt prefix that instructs the LLM to output structured JSON notation events
   - Include a concrete JSON example in the suffix matching the format from CONTEXT.md (measure-level structure, flat note internals, LilyPond-style pitch names)
   - Specify explicit rules: "type": "rest" for rests, dynamic on first note where it changes, articulations/expressions as arrays, pitch as LilyPond-style names (bf4, ef5), duration in quarterLength units
   - Include all instrument names so the LLM outputs one SectionNotation object per instrument
   - Instruct the LLM to output a JSON array of objects, one per instrument

   - Strip markdown code blocks (```json ... ```) -- same pattern as existing extract_lilypond_from_response in fixer.py
   - Try json.loads on the cleaned text first (as array, then as single object wrapped in list)
   - If that fails, try extracting individual JSON objects via regex ({...} blocks with nested braces)
   - Return list of parsed dicts; return empty list on complete failure (never raise)
   - Log a warning on parse failure

Modify src/engrave/generation/pipeline.py to fan out two concurrent LLM requests per section:

1. Add `import asyncio` and `import json` at top. Add import for `build_json_generation_suffix` and `extract_json_from_response` from prompts.

2. Update `generate_section()` signature to return `tuple[str, list[dict] | None, CoherenceState]` -- adding the JSON notation events (or None if JSON generation failed).

3. Inside `generate_section()`:
   - Build the shared prompt prefix using the existing `build_section_prompt()` call
   - Build JSON suffix using `build_json_generation_suffix(instrument_names)`
   - Create two sets of messages:
     - `ly_messages`: the existing prompt (unchanged)
     - `json_messages`: the shared prompt + "\n\n" + JSON suffix
   - Use `asyncio.gather()` to dispatch both `router.complete()` calls concurrently:
     - LilyPond: role="generator", temperature=0.3 (existing)
     - JSON: role="generator", temperature=0.1 (lower for structured output per CONTEXT.md decision)
   - Wrap JSON request in try/except -- if it fails, log warning, set json_result=None, continue with LilyPond
   - Parse JSON response via `extract_json_from_response()`
   - The LilyPond processing path remains UNCHANGED -- extract, parse blocks, build section source, compile-fix loop

4. Update `GenerationResult` dataclass to include:
   - `json_sections: list[list[dict] | None] = field(default_factory=list)` -- collected JSON per section (None for sections where JSON failed)

5. Update `generate_from_midi()`:
   - Collect JSON results alongside section sources
   - After successful section generation, save aligned (LilyPond, JSON) training pair to job directory:
     - `_save_training_pair(ly_source, json_data, section_index, output_dir)` helper
     - Saves as `{output_dir}/training_pairs/section_{N}.json` with fields `ly_source` and `json_notation`
     - If output_dir is not provided (optional parameter), skip saving
   - Add `output_dir: str | None = None` parameter to `generate_from_midi()`
   - Pass collected json_sections to GenerationResult

6. **CRITICAL: Graceful degradation.** If the router doesn't support async (raises NotImplementedError or similar), fall back to sequential requests. If JSON request fails for any reason, the LilyPond path must continue unaffected. The JSON result is always optional.

7. Update src/engrave/generation/__init__.py to export any new public symbols.
