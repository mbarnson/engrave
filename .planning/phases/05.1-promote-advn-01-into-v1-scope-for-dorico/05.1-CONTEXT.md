# Phase 05.1: Promote ADVN-01 into v1 scope for Dorico - Context

**Gathered:** 2026-02-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Generate MusicXML export from Engrave's internal representation — not by converting LilyPond output — for reliable import into Dorico/Sibelius/MuseScore. This promotes ADVN-01 from v2 into v1 scope. The MusicXML is an editable starting point for arrangers, not a finished product; Engrave's PDF remains the beautiful print output.

Key architectural insight: Engrave's pipeline has the musical data (MIDI, key, tempo, sections, instrument assignments, LLM-generated articulations/dynamics) needed to produce clean MusicXML without going through the broken LilyPond-to-MusicXML conversion path.

</domain>

<decisions>
## Implementation Decisions

### Source Representation
- MusicXML must reflect post-LLM enriched content — all the musical intelligence the LLM adds (articulations, dynamics, expressions), not just raw MIDI
- python-ly parses the generated LilyPond document tree to extract structured musical content — its parser works well even though its MusicXML export is broken
- Full engraving data must survive: dynamics, articulations, tempo markings, rehearsal marks, key/time signatures, chord symbols, multi-bar rests
- MusicXML generation is a post-render stage in the existing RenderPipeline — runs after LilyPond files are generated, before ZIP packaging

### Dorico Fidelity
- Target workflow: arrangers import into Dorico to edit and re-engrave, not for playback or round-trip
- Best-effort MusicXML generation — don't dumb down output to avoid edge cases, document known Dorico import limitations in output notes
- Validate against MusicXML 4.0 XSD before packaging; fail gracefully (include .ly but skip .musicxml if invalid)
- Generic standard-compliant MusicXML only, no vendor-specific metadata — Dorico maps instruments from part names and transposition data automatically

### Export Scope
- Single .musicxml file containing all parts (full score) — Dorico imports and auto-creates players/instruments
- Concert pitch throughout — matches Engrave's internal storage convention; Dorico handles transposition natively
- File lives at top level in output ZIP alongside conductor-score.pdf, immediately visible
- MusicXML export on by default — always included in ZIP, disable with --no-musicxml if needed

### Conversion Strategy
- music21 as MusicXML writer — full-featured, well-tested MusicXML 4.0 support, already planned for Phase 9 evaluation
- Direct streaming from python-ly document tree to music21 objects — no intermediate data model; extract an IR later if Phase 9 needs it
- Graceful degradation on parse failures: skip unparseable constructs, include everything that parsed successfully, log warnings
- Testing: MusicXML XSD schema validation + music21 roundtrip (write then re-read, verify note/rhythm/dynamic fidelity); manual Dorico testing outside CI

### Claude's Discretion
- Exact python-ly document tree traversal strategy
- music21 Score construction patterns (Part/Measure/Note hierarchy)
- Warning/logging format for skipped constructs
- How to map LilyPond-specific constructs that have no MusicXML equivalent

</decisions>

<specifics>
## Specific Ideas

- "There is no reliable LilyPond-to-MusicXML export path" — this is a known, long-standing gap in the LilyPond ecosystem. python-ly's musicxml export is buggy, LilyPond has no native MusicXML export, and OCR via Audiveris on rendered PDFs is absurd
- Engrave's advantage: the pipeline starts from MIDI and structured musical data, not from LilyPond. LilyPond is a rendering target, not the source of truth
- Output ZIP should contain: PDFs (beautiful, print-ready), .ly source (for LilyPond users), .musicxml (for Dorico/MuseScore/Sibelius import), and MIDI — covering every downstream workflow
- Arrangers use Engrave's PDF for the beautiful print output, and the MusicXML for the editable score they can tweak in Dorico

</specifics>
