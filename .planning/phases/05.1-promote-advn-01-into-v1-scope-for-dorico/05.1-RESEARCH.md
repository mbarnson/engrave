## User Constraints (from CONTEXT.md)

### Locked Decisions
- MusicXML must reflect post-LLM enriched content -- all the musical intelligence the LLM adds (articulations, dynamics, expressions), not just raw MIDI
- **DO NOT parse LilyPond back via python-ly** -- the original conversion strategy was wrong. Parsing LLM-generated LilyPond is two fragile translation steps (LilyPond text -> python-ly tree -> music21 objects) when the structured musical data exists before LilyPond generation
- The generation pipeline fans out **two parallel requests per section** using the Chatterfart prefix-caching pattern: Request 1 produces LilyPond, Request 2 produces structured JSON notation events -- both from identical prompt context
- With vllm-mlx continuous batching and prefix caching, the shared prompt context (MIDI tokens, audio description, coherence state, RAG examples, user hints) is computed once; the JSON request is nearly free in prompt processing -- only pays generation tokens
- Full engraving data must survive: dynamics, articulations, tempo markings, rehearsal marks, key/time signatures, chord symbols, multi-bar rests
- JSON output is authoritative for MusicXML. LilyPond/JSON disagreements are logged as quality telemetry (feeds TUNE-02) but don't affect MusicXML output
- Measure-level structure with flat note internals -- MusicXML and music21 are fundamentally measure-based
- Per-instrument, per-section JSON matching the LilyPond generation granularity
- JSON format: `{"instrument": "trumpet_1", "key": "bf_major", "time_signature": "4/4", "measures": [{"number": 17, "notes": [{"pitch": "bf4", "beat": 1.0, "duration": 1.0, "articulations": ["marcato"], "dynamic": "f"}, ...]}]}`
- `dynamic` on the first note where it changes (not every note)
- `articulations` and `expressions` as arrays, default to empty/omitted
- `"type": "rest"` distinguishes rests from pitched notes
- Pitch as LilyPond-style names (`bf4`, `ef5`) since the LLM is fluent in that from the parallel LilyPond request; music21 builder maps to `music21.pitch.Pitch("Bb4")` with trivial lookup
- Measure numbers are explicit -- no beat-to-measure arithmetic needed
- Per-section generation fans out 2 requests with shared prefix (Chatterfart pattern)
- Both requests share identical prompt context; only the appended instruction differs
- Request 1 suffix: "Generate LilyPond for this section" (existing behavior)
- Request 2 suffix: "Generate structured JSON notation events for this section"
- Per-section, matching LilyPond -- the prefix cache benefit is per-section; full-score JSON would lose prefix sharing
- Requires backend with batched inference + prefix caching: vllm-mlx (already used for Chatterfart), vllm on CUDA, or any OpenAI-compatible server with prefix caching
- **Graceful degradation on non-batching backends:** the fan-out still works -- just two sequential requests instead of two batched ones, paying full prompt cost twice. Slower and more expensive but functionally correct. Not a hard gate -- just a performance penalty
- music21 as MusicXML writer -- full-featured, well-tested MusicXML 4.0 support, already planned for Phase 9 evaluation
- JSON-to-music21 assembler merges sections the same way the LilyPond assembler does: iterate sections, iterate instruments, append measures to Parts
- Structurally identical to LilyPond assembly, just targeting music21 objects instead of .ly text concatenation
- MusicXML generation integrates into RenderPipeline as a post-generation stage, before ZIP packaging
- Target workflow: arrangers import into Dorico to edit and re-engrave, not for playback or round-trip
- Best-effort MusicXML generation -- don't dumb down output to avoid edge cases, document known Dorico import limitations in output notes
- Validate against MusicXML 4.0 XSD before packaging; fail gracefully (include .ly but skip .musicxml if invalid)
- Generic standard-compliant MusicXML only, no vendor-specific metadata -- Dorico maps instruments from part names and transposition data automatically
- Single .musicxml file containing all parts (full score) -- Dorico imports and auto-creates players/instruments
- Concert pitch throughout -- matches Engrave's internal storage convention; Dorico handles transposition natively
- File lives at top level in output ZIP alongside conductor-score.pdf, immediately visible
- MusicXML export on by default -- always included in ZIP, disable with --no-musicxml if needed
- MusicXML XSD 4.0 schema validation before packaging
- music21 roundtrip: write MusicXML then re-read, verify note/rhythm/dynamic fidelity
- JSON/LilyPond disagreement logging for quality telemetry
- Manual Dorico testing outside CI -- done state: open MusicXML in Dorico, verify correct notes/dynamics/articulations
- **Training data collection:** every parallel fan-out produces aligned (LilyPond, JSON) pairs -- save both outputs as aligned pairs in the job directory. Collection costs nothing and the data is irreplaceable

## Summary

Phase 05.1 adds MusicXML export to the Engrave pipeline by generating structured JSON notation events in parallel with LilyPond output, then building music21 Score objects from that JSON and writing MusicXML 4.0. The key architectural insight is that the LLM already has the full musical context when generating LilyPond -- by fanning out a second request with the same prompt prefix but a different suffix instruction ("generate JSON notation events"), the JSON output captures the same post-LLM enrichment (articulations, dynamics, expressions) at near-zero additional prompt processing cost via prefix caching.

The implementation spans four domains: (1) modifying the generation pipeline to fan out parallel LilyPond+JSON requests per section, (2) building a JSON-to-music21 converter that maps LilyPond-style pitch names and articulation names to music21 objects, (3) assembling per-section music21 data into a complete Score and writing MusicXML, and (4) integrating MusicXML generation and XSD validation into the existing RenderPipeline and ZIP packager. Training data pairs (LilyPond, JSON) are collected as a side effect.

**Primary recommendation:** Use music21 9.x as the MusicXML writer, xmlschema for XSD validation (lxml has known issues with MusicXML 4.0 XSD), Pydantic models for JSON validation, and asyncio.gather for parallel fan-out. Build the pitch-name lookup table as a static mapping from LilyPond names to music21 pitch strings.

### Pattern 1: Parallel Fan-Out with Shared Prefix (Chatterfart Pattern)
**What:** Dispatch two LLM requests per section with identical prompt context but different trailing instructions. The first produces LilyPond (existing behavior), the second produces structured JSON. With prefix caching, the shared KV cache is computed once.
**When to use:** Every section generation call.
**Example:**
```python
# In generation/pipeline.py, inside the section generation loop
import asyncio

async def generate_section_with_json(
    section_midi: dict[str, str],
    coherence: CoherenceState,
    rag_examples: list[str],
    template: str,
    instrument_names: list[str],
    router: InferenceRouter,
    compiler: LilyPondCompiler,
) -> tuple[str, dict, CoherenceState]:
    """Fan out LilyPond + JSON generation for one section."""

    # Build the shared prompt prefix (identical for both requests)
    base_prompt = build_section_prompt(section_midi, coherence, rag_examples, template)

    # Two suffixes: LilyPond generation and JSON generation
    ly_messages = [{"role": "user", "content": base_prompt}]
    json_suffix = build_json_generation_suffix(instrument_names)
    json_messages = [{"role": "user", "content": base_prompt + "\n\n" + json_suffix}]

    # Fan out both requests concurrently
    ly_task = router.complete(role="generator", messages=ly_messages, temperature=0.3)
    json_task = router.complete(role="generator", messages=json_messages, temperature=0.1)

    ly_response, json_response = await asyncio.gather(ly_task, json_task)

    # Parse and validate JSON
    notation_events = parse_and_validate_json(json_response)

    # Save aligned pair for training data
    save_training_pair(ly_response, json_response, section_index)

    # Continue with existing LilyPond processing...
    return ly_source, notation_events, updated_coherence
```

### Pattern 2: JSON-to-music21 Builder (Bottom-Up Construction)
**What:** Build music21 objects bottom-up: Pitch -> Note -> Measure -> Part -> Score. Each JSON notation event maps to a music21 object through a deterministic lookup.
**When to use:** After collecting all per-section JSON outputs, assemble into a complete Score.
**Example:**
```python
# In musicxml/builder.py
import music21

from engrave.musicxml.pitch_map import ly_pitch_to_m21

    """Convert a single JSON note event to a music21 Note or Rest."""
    if note_event.type == "rest":
        r = music21.note.Rest()
        r.quarterLength = note_event.duration
        return r

    n = music21.note.Note()
    n.pitch = music21.pitch.Pitch(ly_pitch_to_m21(note_event.pitch))
    n.quarterLength = note_event.duration

    for art_name in (note_event.articulations or []):
        art_cls = ARTICULATION_MAP.get(art_name)
        if art_cls:
            n.articulations.append(art_cls())

    return n

    """Convert a JSON measure to a music21 Measure."""
    m = music21.stream.Measure(number=measure_number)

        m.append(build_note(note_event))
```

### Pattern 3: Pydantic Models for LLM JSON Validation
**What:** Define strict Pydantic models matching the JSON notation event schema. Validate LLM output immediately after parsing. Invalid JSON triggers a warning (MusicXML skipped for that section) but does not halt generation.
**When to use:** Every JSON response from the LLM.
**Example:**
```python
# In musicxml/models.py
from pydantic import BaseModel, field_validator

class NoteEvent(BaseModel):
    pitch: str | None = None  # LilyPond-style: "bf4", "ef5"
    type: str | None = None   # "rest" for rests, None for pitched notes
    beat: float
    duration: float
    articulations: list[str] | None = None
    expressions: list[str] | None = None
    dynamic: str | None = None

    @field_validator("pitch")
    @classmethod
    def validate_pitch(cls, v):
        if v is not None and not re.match(r"^[a-g](f|s|es|is|ff|ss)?[0-9]$", v):
            raise ValueError(f"Invalid LilyPond pitch: {v}")
        return v

class MeasureData(BaseModel):
    number: int
    notes: list[NoteEvent]

class SectionNotation(BaseModel):
    instrument: str
    key: str | None = None
    time_signature: str | None = None
    measures: list[MeasureData]
```

### Anti-Patterns to Avoid
- **Parsing LilyPond to get MusicXML:** The user explicitly locked this out. Never convert LilyPond text back through python-ly or any parser. The JSON is the authoritative source.
- **Blocking LilyPond on JSON success:** JSON validation failure must not prevent LilyPond compilation. The LilyPond path is independent and must always succeed regardless of JSON issues.
- **Full-score JSON generation:** Do not generate JSON for the entire score in one LLM call. Per-section generation preserves prefix-cache benefits and matches the existing LilyPond generation granularity.
- **Vendor-specific MusicXML extensions:** Do not add Dorico-specific metadata. Standard MusicXML 4.0 only. Dorico handles instrument mapping from part names.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| MusicXML file generation | Custom XML element tree builder | music21 Score.write('musicxml') | MusicXML has complex element ordering, attribute requirements, and DTD conformance. music21 handles makeNotation(), duration splitting, voice assignment, and correct XML namespace declarations. |
| XSD validation | Custom validation logic or lxml XMLSchema | xmlschema.XMLSchema.validate() | MusicXML 4.0 XSD has nested imports and external entity references that break lxml. xmlschema handles these correctly. |
| LilyPond pitch to MIDI/music21 mapping | Regex-based octave arithmetic | Static lookup table + pitch.Pitch() constructor | Edge cases: double-sharps, double-flats, enharmonic spelling, octave boundaries (b3 vs c4). A lookup table with music21 verification is reliable. |
| JSON schema validation | Manual if/else checking of LLM output | Pydantic model_validate() | LLM output is unpredictable. Pydantic gives typed validation with clear error messages, automatic type coercion where safe, and serialization for telemetry logging. |

**Key insight:** music21's MusicXML export handles dozens of tricky details -- tuplet notation, grace notes, tied note splitting across barlines, beam group assignment, voice numbering, backup/forward elements -- that would take weeks to build and debug from scratch. The library has 15+ years of development behind its MusicXML export.

## Common Pitfalls

### Pitfall 1: music21 Pitch Name Format Mismatch
**What goes wrong:** LilyPond uses `bf` for B-flat, `ef` for E-flat, `fis` for F-sharp. music21 uses `B-` for B-flat, `E-` for E-flat, `F#` for F-sharp. Passing LilyPond names directly to music21.pitch.Pitch() will fail or produce wrong pitches.
**Why it happens:** Different naming conventions across music notation ecosystems.
**How to avoid:** Build an explicit lookup table mapping LilyPond pitch names (without octave) to music21 pitch name strings. The table is small and static:
```python
_LY_TO_M21 = {
    "c": "C", "d": "D", "e": "E", "f": "F", "g": "G", "a": "A", "b": "B",
    "cf": "C-", "df": "D-", "ef": "E-", "ff": "F-", "gf": "G-", "af": "A-", "bf": "B-",
    "cs": "C#", "ds": "D#", "es": "E#", "fs": "F#", "gs": "G#", "as": "A#", "bs": "B#",
    "cis": "C#", "dis": "D#", "eis": "E#", "fis": "F#", "gis": "G#", "ais": "A#", "bis": "B#",
    "ces": "C-", "des": "D-", "ees": "E-", "fes": "F-", "ges": "G-", "aes": "A-", "bes": "B-",
}
```
Then extract the octave number from the trailing digit and construct: `pitch.Pitch(f"{_LY_TO_M21[name]}{octave}")`.
**Warning signs:** Pitches that are a semitone off, or pitch.Pitch() constructor ValueError exceptions.

### Pitfall 2: music21 Duration as quarterLength, Not Beat Count
**What goes wrong:** JSON format uses `"duration": 1.0` meaning one beat (one quarter note in 4/4). music21 measures duration in `quarterLength` where 1.0 = one quarter note. In 4/4 time these align, but in 6/8 time a "beat" could be a dotted quarter (1.5 quarterLengths).
**Why it happens:** Ambiguity between "beat" and "quarter note" in compound meters.
**How to avoid:** Define the JSON `duration` field as quarterLength explicitly (not beats). The JSON format example in CONTEXT.md uses 1.0 for a quarter note and 0.5 for an eighth note, which IS quarterLength notation. Document this clearly in the schema and prompt suffix.
**Warning signs:** Notes that are twice as long or half as long as expected in the MusicXML output.

### Pitfall 3: LLM JSON Output Not Valid JSON
**What goes wrong:** The LLM wraps JSON in markdown code blocks (` ```json ... ``` `), adds commentary before/after the JSON, or generates trailing commas.
**Why it happens:** LLMs are trained on conversational text that includes code formatting.
**How to avoid:** Strip markdown code blocks before parsing (same pattern used in `extract_lilypond_from_response`). Use a lenient JSON extraction function. When json.loads fails, try extracting the first `{...}` or `[...]` block. If all parsing fails, log the raw response and skip MusicXML for this section.
**Warning signs:** json.JSONDecodeError on LLM responses that look correct when read by a human.

### Pitfall 4: music21 makeNotation() Altering Intended Output
**What goes wrong:** music21's `makeNotation()` (called automatically during MusicXML export) adds time signatures, key signatures, clefs, or splits notes across barlines in ways that differ from the intended notation.
**Why it happens:** music21 attempts to create "correct" notation which may not match the LLM's intent.
**How to avoid:** Insert explicit TimeSignature, KeySignature, and Clef objects at the start of each Part before calling `write('musicxml')`. When music21 sees these already present, it respects them rather than inferring. For note splitting, ensure note durations never exceed the remaining space in a measure.
**Warning signs:** Extra time signatures appearing, notes split into tied pairs unexpectedly.

### Pitfall 5: Dorico MusicXML Import Quirks
**What goes wrong:** Dorico may display articulations differently, lose text expressions, or reorganize tuplets on MusicXML import.
**Why it happens:** Known limitations in Dorico's MusicXML import -- documented on Steinberg forums. MusicXML is a lossy interchange format; not all notation concepts round-trip perfectly.
**How to avoid:** This is a "best-effort" target per user decision. Test with real Dorico imports but don't gate on Dorico fidelity. Common issues: dynamics not recognized for playback (fine for editing workflow), grace notes before tuplets causing note loss, chord symbol positioning.
**Warning signs:** Missing articulations in Dorico, unexpected rests, misaligned dynamics.

### Pitfall 6: Key Signature Mapping from LilyPond to music21
**What goes wrong:** JSON `"key": "bf_major"` needs to become `music21.key.Key('B-')`. The mapping between LilyPond key naming and music21 key naming requires explicit translation.
**Why it happens:** LilyPond uses `\key bes \major`, JSON stores `bf_major`, music21 uses `Key('B-')`.
**How to avoid:** Build a key lookup table similar to the pitch lookup. Parse the underscore-separated format (`bf_major` -> pitch=`bf`, mode=`major`) and map through the same pitch lookup table, then construct `key.Key(f"{mapped_pitch} {mode}")`.
**Warning signs:** Wrong key signatures in MusicXML, or key.Key constructor errors.

### LilyPond Pitch to music21 Pitch Conversion
```python
# Source: music21 official docs (pitch module) + LilyPond naming conventions
import re
import music21

# Static lookup: LilyPond note name (without octave) -> music21 pitch name
_LY_NOTE_TO_M21: dict[str, str] = {
    "c": "C", "d": "D", "e": "E", "f": "F", "g": "G", "a": "A", "b": "B",
    "cf": "C-", "df": "D-", "ef": "E-", "ff": "F-", "gf": "G-", "af": "A-", "bf": "B-",
    "cs": "C#", "ds": "D#", "es": "E#", "fs": "F#", "gs": "G#", "as": "A#", "bs": "B#",
    "cis": "C#", "dis": "D#", "eis": "E#", "fis": "F#", "gis": "G#", "ais": "A#", "bis": "B#",
    "ces": "C-", "des": "D-", "ees": "E-", "fes": "F-", "ges": "G-", "aes": "A-", "bes": "B-",
}

_LY_PITCH_RE = re.compile(r"^([a-g](?:es|is|f|s)?)(\d)$")

def ly_pitch_to_m21(ly_pitch: str) -> str:
    """Convert LilyPond-style pitch (e.g. 'bf4') to music21 pitch string (e.g. 'B-4').

    Args:
        ly_pitch: LilyPond pitch name with octave digit, e.g. "bf4", "fis5", "c3".

    Returns:
        music21-compatible pitch string, e.g. "B-4", "F#5", "C3".

    Raises:
        ValueError: If the pitch name is not recognized.
    """
    match = _LY_PITCH_RE.match(ly_pitch)
    if not match:
        raise ValueError(f"Cannot parse LilyPond pitch: {ly_pitch!r}")

    note_name, octave = match.group(1), match.group(2)
    m21_name = _LY_NOTE_TO_M21.get(note_name)
    if m21_name is None:
        raise ValueError(f"Unknown LilyPond note name: {note_name!r}")

    return f"{m21_name}{octave}"
```

### music21 Score Construction
```python
# Source: music21 usersGuide chapters 4, 15, 32
import music21

def build_part(instrument_name: str, measures: list, key_str: str, time_sig: str, tempo_bpm: int) -> music21.stream.Part:
    """Build a music21 Part from JSON section data."""
    part = music21.stream.Part()
    part.partName = instrument_name

    # Add key signature and time signature to first measure
    first_measure = music21.stream.Measure(number=measures[0]["number"])
    first_measure.append(music21.key.Key(key_str))

    ts_parts = time_sig.split("/")
    first_measure.append(music21.meter.TimeSignature(time_sig))

    # Add tempo marking
    mm = music21.tempo.MetronomeMark(number=tempo_bpm)
    first_measure.append(mm)

    # Add notes to first measure
    for note_data in measures[0]["notes"]:
        first_measure.append(build_note(note_data))

    part.append(first_measure)

    # Remaining measures
    for m_data in measures[1:]:
        m = music21.stream.Measure(number=m_data["number"])
        for note_data in m_data["notes"]:
            m.append(build_note(note_data))
        part.append(m)

    return part
```

### MusicXML Export with Validation
```python
# Source: music21 m21ToXml module + xmlschema docs
import tempfile
from pathlib import Path
import xmlschema

def write_and_validate_musicxml(
    score: music21.stream.Score,
    output_path: Path,
    xsd_path: Path,
) -> tuple[bool, str]:
    """Write music21 Score to MusicXML and validate against XSD.

    Returns:
        (is_valid, error_message) tuple.
    """
    # Write MusicXML
    score.write("musicxml", fp=str(output_path))

    # Validate against XSD
    schema = xmlschema.XMLSchema(str(xsd_path))
    try:
        schema.validate(str(output_path))
        return True, ""
    except xmlschema.XMLSchemaValidationError as e:
        return False, str(e)
```

### JSON Extraction from LLM Response
```python
# Pattern from existing extract_lilypond_from_response in fixer.py
import json
import re

def extract_json_from_response(response: str) -> list[dict]:
    """Extract JSON notation events from LLM response.

    Handles markdown code blocks, surrounding commentary, and
    multiple JSON objects (one per instrument).
    """
    # Strip markdown code blocks
    code_block = re.search(r"```(?:json)?\s*\n(.*?)\n```", response, re.DOTALL)
    if code_block:
        text = code_block.group(1)
    else:
        text = response

    # Try parsing as JSON array first
    try:
        parsed = json.loads(text)
        if isinstance(parsed, list):
            return parsed
        return [parsed]
    except json.JSONDecodeError:
        pass

    # Try extracting individual JSON objects
    objects = []
    for match in re.finditer(r"\{[^{}]*(?:\{[^{}]*\}[^{}]*)*\}", text, re.DOTALL):
        try:
            obj = json.loads(match.group())
            objects.append(obj)
        except json.JSONDecodeError:
            continue

    return objects
```

- python-ly musicxml export: Buggy, incomplete, known gap in LilyPond ecosystem. Explicitly excluded by user decision.

## Open Questions

1. **music21 MusicXML version header**
   - What we know: music21 writes MusicXML but the exact version declaration in the output XML header needs verification
   - What's unclear: Whether music21 9.x writes MusicXML 4.0 or 3.1 by default
   - Recommendation: Test during implementation. If it writes 3.1, check if there's a configuration option or if 3.1 is sufficient for Dorico import. Dorico reads both versions.

2. **JSON generation prompt calibration**
   - What we know: The prompt suffix needs to instruct the LLM to output the exact JSON format
   - What's unclear: How reliably the LLM will produce well-formed JSON matching the schema on first attempt
   - Recommendation: Use lower temperature (0.1) for JSON generation vs 0.3 for LilyPond. Include a concrete JSON example in the prompt suffix. Pydantic validation catches malformed output; graceful degradation skips MusicXML for that section.

3. **Concurrent request fan-out with LiteLLM**
   - What we know: asyncio.gather can dispatch two router.complete() calls concurrently. vllm-mlx with prefix caching batches them efficiently.
   - What's unclear: Whether LiteLLM's acompletion correctly handles two concurrent calls to the same endpoint without issues (connection pooling, rate limiting)
   - Recommendation: Test empirically. If concurrent calls fail, fall back to sequential dispatch. The user decision explicitly allows graceful degradation for non-batching backends.

4. **MusicXML 4.0 XSD file distribution**
   - What we know: XSD files are available from W3C GitHub (w3c/musicxml) as a downloadable ZIP
   - What's unclear: Whether to vendor the XSD files in the repo or download at runtime
   - Recommendation: Vendor the XSD files in `src/engrave/musicxml/schema/`. They are stable, small, and required for offline validation. Pin to MusicXML 4.0 release version.

5. **Training data pair storage format**
   - What we know: User wants aligned (LilyPond, JSON) pairs saved in the job directory
   - What's unclear: Exact file naming and directory structure for training data
   - Recommendation: Save as `{job_dir}/training_pairs/section_{N}_{instrument}.json` with both `ly_source` and `json_notation` fields in each file.
