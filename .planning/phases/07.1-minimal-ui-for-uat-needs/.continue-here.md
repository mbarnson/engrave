## Root cause bug identified (fix this first!)

The LLM returns the full template including `piano = { ... }` wrapper, but
`_build_section_source()` in pipeline.py ALSO wraps the content in `varName = { ... }`.
Result: duplicate variable declaration → LilyPond compile error "not a note name: piano".

The fix: `parse_instrument_blocks()` or `extract_lilypond_from_response()` must strip
the template wrapper so only the inner music content reaches `build_instrument_variable()`.
Look at `src/engrave/generation/templates.py:parse_instrument_blocks()`.

Verbose log proving this is at: `jobs/e2e-test/bigband3/generate.log`
The error: `Fix loop attempt 1: 3 errors. First: not a note name: piano`
The full source shows `piano = {` appearing twice (lines 5 and 9).
