# Phase 07.1: Minimal UI for UAT Needs - Context

**Gathered:** 2026-02-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Build an automated smoke test harness and a minimal web UI so the full Engrave pipeline can be validated end-to-end before UAT with Sam. The smoke harness catches crashes, malformed output, and pipeline failures automatically. The web UI gives a non-technical user a way to exercise the pipeline without CLI. A markdown test plan, pre-validated by the harness, gives Sam a structured way to evaluate musical quality.

This phase does NOT include: quality evaluation against reference scores (Phase 9), YouTube URL ingestion in the UI, user accounts, job history, output format toggles, or any Phase 10 Web UI polish.

</domain>

<decisions>
## Implementation Decisions

### Smoke Test Harness

- CLI command: `engrave smoke-test ./test-fixtures/` — takes a directory of test inputs
- Discovers inputs by file extension: `.wav`/`.mp3`/`.flac`/`.aiff` run audio pipeline, `.mid` runs MIDI-only pipeline
- Runs both input paths: audio-in (process-audio → generate → render) and MIDI-only (generate → render). MIDI-only is the fast debugging loop — when audio-in fails, compare with MIDI-in to isolate whether it's audio pipeline or generation
- Test inputs are NOT shipped with the harness. User populates the test directory with real audio/MIDI files. A README notes where to get bTd MIDIs and what formats are supported
- Sam can later drop his own tracks into the directory and run the harness as a "does Engrave work on my music" tool

**Checks (per input file):**
1. No exceptions — pipeline completes without crashing
2. Compilable LilyPond — the .ly source compiles without errors
3. Valid PDFs — conductor score and all parts produce PDF output
4. Valid MusicXML — passes XSD 4.0 validation, music21 can re-read it without errors
5. All parts present — ZIP contains expected number of files (conductor + N parts + .ly + .musicxml + MIDI)
6. Correct transpositions — Bb trumpet part in Bb, Eb alto in Eb, etc.
7. Note count > 0 per non-drum part — catches "valid but empty" failure mode
8. PDF file size > minimum threshold — empty part PDF is ~20KB, real one is 50KB+; catches LLM producing nothing useful
9. ZIP file count matches expected (conductor score + N parts + .ly sources + .musicxml + MIDI)

**Explicitly skipped checks:** tempo marking presence, empty page detection — those are quality concerns dependent on input.

**Output:** Terminal (human-readable pass/fail) + JSON (structured, for coding agent to consume when debugging failures)

### Minimal Web UI

- One HTML page, one FastAPI endpoint. No framework beyond vanilla JS or htmx
- Direct Python API calls — FastAPI endpoint calls `AudioPipeline.process()`, `generate_from_midi()`, `RenderPipeline.render()` directly. NOT shelling out to CLI. The pipeline is async, FastAPI is async, everything lines up naturally
- Inputs: file upload (MIDI or audio), hints textarea, Engrave button
- Status: simple polling (POST /engrave returns job ID, JS polls GET /status/{id} every 3 seconds). No SSE, no WebSocket — one user at a time on local machine, polling overhead is invisible
- No real-time stage progress (would require pipeline to emit structured progress events, which it doesn't). Just "Processing..." indicator
- Output: download ZIP link when complete
- NOT included: YouTube URL field, accounts, job history, output format toggles

### UAT Test Plan

- Markdown checklist (TESTPLAN.md) with steps and expected results
- Each step pre-validated by running the smoke harness — Sam shouldn't discover crashes
- Sam's job is musical quality judgment: "open trumpet 1 part, can you sight-read it? yes/no, notes:"
- Sam can scribble in the checklist or just text feedback — not a QA form

### Phase Sequencing

- Depends on Phase 6 (current), NOT Phase 7. Pipeline already runs end-to-end
- Build harness first to find crash-paste-fix bugs on stable ground before Phase 7's per-section joint generation refactor
- Phase 7 is purely additive from harness perspective — all checks apply identically before and after
- Execution order: Phase 6 → 7.1 (harness + UI) → Phase 7 → re-run harness → UAT with Sam

</decisions>

<specifics>
## Specific Ideas

- The smoke harness is not a test suite — it's a "does Engrave work on my music" tool. It discovers inputs by extension and runs them through the appropriate pipeline path. No test framework, no fixtures, no assertions in the pytest sense. Just a CLI command that reports what worked and what didn't.
- JSON output is specifically for the coding agent: when something fails, paste the JSON, not a screenshot of terminal colors
- The web UI is the same API the CLI uses — FastAPI calls the same functions the CLI commands wrap. No new integration code, just a different entry point.
- "Valid but empty" is the predicted most common failure mode — a PDF that compiles fine but has 32 bars of rest because the LLM produced nothing useful

</specifics>
