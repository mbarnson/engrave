## Summary

Phase 07.1 builds three independent deliverables: (1) an automated smoke test harness that discovers audio/MIDI test inputs and runs them through the full Engrave pipeline, performing 9 structural checks per file; (2) a minimal single-page web UI using FastAPI that gives Sam a browser-based way to upload files, add hints, and download output ZIPs without touching the CLI; and (3) a markdown UAT test plan pre-validated by the harness.

The codebase already has all the pipeline components needed. `AudioPipeline.process()` handles audio-in, `generate_from_midi()` handles MIDI-to-LilyPond generation, and `RenderPipeline.render()` handles rendering and ZIP packaging. The smoke harness orchestrates these same functions and adds structural validation checks on the output. The web UI is a thin FastAPI layer over the same function calls the CLI uses.

FastAPI is not currently a project dependency but is the user's stated choice for the web framework. It must be added to `pyproject.toml`. The HTML/JS frontend is intentionally minimal -- one page with file upload, a textarea, a button, and a status/download area. htmx provides the cleanest polling pattern (load polling with `hx-trigger="every 3s"` and 286 stop code) without requiring any JavaScript framework.

**Primary recommendation:** Build the smoke test harness first -- it validates that the pipeline works end-to-end and catches crash/malformation bugs. Then build the web UI as a separate thin layer. Write TESTPLAN.md last, after the harness confirms which test inputs pass.

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| htmx polling | Vanilla JS fetch + setInterval | htmx eliminates all client-side polling code. `hx-get="/status/{id}" hx-trigger="every 3s"` replaces 15+ lines of JavaScript. HTTP 286 stop code is cleaner than client-side state management |

| Jinja2 template | Inline HTML string | For a single page, inline HTML in an `HTMLResponse` or a single template file is simpler than setting up Jinja2. Recommend: one `templates/index.html` file served by `Jinja2Templates` -- cleaner separation, easy to iterate on styling |

### Pattern 1: Smoke Test Runner (Input Discovery + Pipeline Orchestration)
**What:** A runner class that discovers test inputs by extension, routes them through the appropriate pipeline path, and collects per-input results.
**When to use:** The `engrave smoke-test` CLI command.

```python
# src/engrave/smoke/runner.py
from __future__ import annotations
import asyncio
import logging
import time
from dataclasses import dataclass, field
from pathlib import Path

logger = logging.getLogger(__name__)

AUDIO_EXTENSIONS = {".wav", ".mp3", ".flac", ".aiff"}
MIDI_EXTENSIONS = {".mid", ".midi"}

@dataclass
class InputResult:
    """Result for a single test input file."""
    input_path: Path
    pipeline_path: str  # "audio" or "midi"
    checks: dict[str, CheckResult] = field(default_factory=dict)
    elapsed_seconds: float = 0.0
    error: str | None = None  # Pipeline-level exception if any

@dataclass
class CheckResult:
    """Result of a single check on a pipeline output."""
    name: str
    passed: bool
    message: str = ""
    details: dict = field(default_factory=dict)

@dataclass
class SmokeResult:
    """Aggregate result of a full smoke test run."""
    inputs: list[InputResult] = field(default_factory=list)
    total_passed: int = 0
    total_failed: int = 0
    total_errors: int = 0
    elapsed_seconds: float = 0.0

def discover_inputs(test_dir: Path) -> list[tuple[Path, str]]:
    """Discover test input files by extension.

    Returns list of (path, pipeline_type) tuples where
    pipeline_type is "audio" or "midi".
    """
    inputs = []
    for f in sorted(test_dir.iterdir()):
        if f.is_file():
            ext = f.suffix.lower()
            if ext in AUDIO_EXTENSIONS:
                inputs.append((f, "audio"))
            elif ext in MIDI_EXTENSIONS:
                inputs.append((f, "midi"))
    return inputs
```

### Pattern 3: FastAPI Web UI with Polling
**What:** A minimal FastAPI app with 3 endpoints and one HTML page. File upload saves to disk in the request handler, kicks off async pipeline processing, returns a job ID. Client polls for status.
**When to use:** The `engrave serve` CLI command.

```python
# src/engrave/web/app.py
from __future__ import annotations
import asyncio
import uuid
import shutil
from pathlib import Path
from enum import Enum

from fastapi import FastAPI, UploadFile, Form, Request
from fastapi.responses import FileResponse, HTMLResponse
from fastapi.staticfiles import StaticFiles
from fastapi.templating import Jinja2Templates

class JobStatus(str, Enum):
    PENDING = "pending"
    PROCESSING = "processing"
    COMPLETE = "complete"
    FAILED = "failed"

# In-memory job store (single user, local machine)
jobs: dict[str, dict] = {}

app = FastAPI(title="Engrave", docs_url=None, redoc_url=None)
templates = Jinja2Templates(directory=str(Path(__file__).parent / "templates"))

@app.get("/", response_class=HTMLResponse)
async def index(request: Request):
    return templates.TemplateResponse("index.html", {"request": request})

@app.post("/engrave")
async def start_engrave(
    file: UploadFile,
    hints: str = Form(""),
):
    """Accept file upload, save to disk, start pipeline in background."""
    job_id = str(uuid.uuid4())[:8]
    job_dir = Path("jobs") / f"web_{job_id}"
    job_dir.mkdir(parents=True, exist_ok=True)

    # Save uploaded file to disk BEFORE starting background task
    # (UploadFile is closed after request handler returns)
    input_path = job_dir / file.filename
    with open(input_path, "wb") as f:
        shutil.copyfileobj(file.file, f)

    jobs[job_id] = {
        "status": JobStatus.PROCESSING,
        "input_path": input_path,
        "hints": hints,
        "zip_path": None,
        "error": None,
    }

    # Start pipeline as asyncio task
    asyncio.create_task(_run_pipeline(job_id, input_path, hints, job_dir))

    return {"job_id": job_id}

@app.get("/status/{job_id}")
async def get_status(job_id: str):
    """Return job status. htmx polls this endpoint."""
    job = jobs.get(job_id)
    if job is None:
        return HTMLResponse("<p>Job not found</p>", status_code=404)

    if job["status"] == JobStatus.PROCESSING:
        return HTMLResponse(
            f'<div id="status" hx-get="/status/{job_id}" '
            f'hx-trigger="every 3s" hx-swap="outerHTML">'
            f'<p>Processing...</p></div>'
        )
    elif job["status"] == JobStatus.COMPLETE:
        # Return 286 to stop htmx polling, show download link
        return HTMLResponse(
            f'<div id="status"><p>Complete!</p>'
            f'<a href="/download/{job_id}">Download ZIP</a></div>',
            status_code=286,
        )
    else:
        return HTMLResponse(
            f'<div id="status"><p>Failed: {job["error"]}</p></div>',
            status_code=286,
        )

@app.get("/download/{job_id}")
async def download(job_id: str):
    """Serve the output ZIP file."""
    job = jobs.get(job_id)
    if job is None or job["zip_path"] is None:
        return HTMLResponse("Not found", status_code=404)
    return FileResponse(job["zip_path"], filename=Path(job["zip_path"]).name)

async def _run_pipeline(
    job_id: str, input_path: Path, hints: str, job_dir: Path
) -> None:
    """Execute the full Engrave pipeline for a web upload."""
    try:
        ext = input_path.suffix.lower()
        if ext in {".wav", ".mp3", ".flac", ".aiff"}:
            # Audio pipeline -> generate -> render
            ...
        elif ext in {".mid", ".midi"}:
            # MIDI-only: generate -> render
            ...
        jobs[job_id]["status"] = JobStatus.COMPLETE
        jobs[job_id]["zip_path"] = str(zip_path)
    except Exception as exc:
        jobs[job_id]["status"] = JobStatus.FAILED
        jobs[job_id]["error"] = str(exc)
```

### Anti-Patterns to Avoid
- **Shelling out to CLI from FastAPI:** The user decision is explicit -- FastAPI calls pipeline Python APIs directly (`AudioPipeline.process()`, `generate_from_midi()`, `RenderPipeline.render()`). No subprocess calls to the `engrave` CLI.
- **Using FastAPI BackgroundTasks with UploadFile:** Since FastAPI 0.106.0, UploadFile is closed before BackgroundTasks execute. Must save file to disk in the request handler, then process the saved file in the background task.
- **Building a real job queue:** Single user on local machine. An in-memory `dict[str, dict]` is sufficient. No Redis, no Celery, no database.
- **Adding WebSocket or SSE for status updates:** Explicitly deferred per CONTEXT.md. Simple polling every 3 seconds is appropriate for one user on localhost.
- **Making the smoke harness a test suite (pytest):** The harness is NOT a test framework. It is a CLI tool that discovers real user files and reports what worked. No `assert`, no fixtures, no test isolation.
- **Validating LilyPond by re-compiling from ZIP:** The .ly files in the ZIP were already compiled by the pipeline. For check 2, verify that compilation succeeded by checking that corresponding PDFs exist, not by re-running LilyPond.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| MusicXML XSD validation | Custom XML validation | Existing `validate_musicxml()` in `src/engrave/musicxml/validator.py` | Already implemented and tested in Phase 5.1 |

| File upload handling | Custom multipart parser | FastAPI `UploadFile` + `python-multipart` | Standard FastAPI pattern |
| Polling mechanism | Custom JavaScript fetch loop | htmx `hx-trigger="every 3s"` with 286 stop | htmx handles polling lifecycle including stop-on-completion |
| Terminal formatting | Custom ANSI codes | Rich console + Rich table | Already used throughout the CLI |
| ZIP inspection | Custom archive parser | stdlib `zipfile.ZipFile` | Python standard library handles all ZIP inspection needs |

**Key insight:** The smoke harness and web UI are thin orchestration layers over existing pipeline components. No new ML models, no new data formats, no new protocols. The technical risk is near zero -- it is plumbing and presentation.

## Common Pitfalls

### Pitfall 1: UploadFile Closed Before Background Processing
**What goes wrong:** The uploaded file handle is closed by FastAPI/Starlette after the request handler returns. Background tasks that try to read the UploadFile get an empty or closed file.
**Why it happens:** FastAPI 0.106.0+ automatically closes UploadFile after the response is sent. This is a deliberate resource management change.
**How to avoid:** Save the uploaded file to disk in the request handler using `shutil.copyfileobj(file.file, destination)`. The background task reads from the saved disk file, not the UploadFile.
**Warning signs:** Empty input files in job directories; pipeline errors about zero-byte input files.

### Pitfall 2: Async Pipeline in FastAPI Event Loop
**What goes wrong:** `generate_from_midi()` is an async function. Calling it directly from a `BackgroundTask` (which runs in a thread pool for sync functions) causes event loop errors.
**Why it happens:** FastAPI BackgroundTasks runs sync functions in a thread pool. Async functions need to be awaited in the existing event loop.
**How to avoid:** Use `asyncio.create_task()` instead of `BackgroundTasks.add_task()` for the pipeline execution. Since the pipeline functions are async and FastAPI runs on an async event loop, `create_task` is the natural fit. The request handler returns the job ID immediately while the task runs in the background.
**Warning signs:** `RuntimeError: There is no current event loop in thread` or `RuntimeError: asyncio.run() cannot be called from a running event loop`.

### Pitfall 3: Smoke Harness Fails on Missing LLM Provider
**What goes wrong:** The smoke harness tries to run `generate_from_midi()` which requires a configured LLM provider. If no provider is configured (no API keys, no local LMStudio), the harness crashes on every input.
**How to avoid:** The harness should check LLM provider connectivity (similar to `engrave check all`) before running any inputs. If no provider is available, report it clearly and exit early rather than failing on every file.
**Warning signs:** All inputs fail with `ProviderError` or `RoleNotFoundError`.

### Pitfall 4: MusicXML Not Present in ZIP (Phase 5.1 Optional)
**What goes wrong:** Check 4 (valid MusicXML) fails because the ZIP does not contain a `.musicxml` file. MusicXML generation is optional in the pipeline -- if JSON notation fails, MusicXML is skipped gracefully.
**Why it happens:** The MusicXML path depends on successful JSON fan-out from `generate_from_midi()`. If the LLM fails to produce valid JSON, MusicXML is omitted from the ZIP.
**How to avoid:** Make check 4 conditional: if no `.musicxml` file exists in the ZIP, report as "SKIPPED (MusicXML not generated)" rather than "FAILED." The absence of MusicXML is a known degradation mode, not a failure.
**Warning signs:** Every input reports MusicXML validation failure even though the pipeline completed successfully.

### Pitfall 5: Pipeline Timeout for Long Audio Files
**What goes wrong:** The web UI appears to hang indefinitely because a long audio file takes 10+ minutes through separation + transcription + generation + rendering.
**Why it happens:** No timeout on the pipeline execution. The 3-second polling keeps checking but the user has no feedback about progress.
**How to avoid:** Set a configurable timeout on pipeline execution (default: 30 minutes). If exceeded, mark the job as failed with a timeout error. Consider adding a rough elapsed-time display to the status response (e.g., "Processing... (2m 15s)") so the user knows something is happening.
**Warning signs:** Web UI shows "Processing..." forever; user refreshes page and loses track of the job.

### Pitfall 6: Transposition Check Complexity
**What goes wrong:** Check 6 (correct transpositions) is harder than expected because the MusicXML `<transpose>` element may not be present if the LLM generated content without explicit transposition metadata.
**Why it happens:** The `RenderPipeline` applies transposition via LilyPond `\transpose` commands at render time. The resulting MusicXML (generated from JSON, not from rendered LilyPond) may not include transposition information if the JSON builder does not embed `<transpose>` elements.
**How to avoid:** Two strategies: (a) Check transposition from the MusicXML `<transpose>` element if present. (b) Alternatively, check that the key signature of a transposing part differs from the concert key by the expected interval. If neither is feasible, downgrade check 6 to "manual verification" and flag it in TESTPLAN.md for Sam.
**Warning signs:** Transposition checks pass vacuously because no `<transpose>` elements exist; false sense of correctness.

### Pitfall 7: Concurrent Job Corruption in Web UI
**What goes wrong:** If two browser tabs submit files simultaneously, the in-memory `jobs` dict and shared `jobs/` directory could have naming conflicts.
**Why it happens:** Although the CONTEXT.md says "one user at a time on local machine," a user might accidentally submit twice.
**How to avoid:** Each job gets a UUID-based directory name (`web_{uuid[:8]}`). The in-memory dict is keyed by job ID. No shared state between jobs. This naturally handles concurrent submissions even though it is not a design goal.
**Warning signs:** ZIP files from different inputs appear in the same job directory; status polling returns wrong job's status.

### music21 Note Count Verification
```python
import music21

def count_notes_per_part(musicxml_path: Path) -> dict[str, int]:
    """Count notes per part in a MusicXML file using music21.

    Returns dict mapping part name to note count.
    Rests are excluded from count.
    """
    score = music21.converter.parse(str(musicxml_path))
    counts = {}
    for part in score.parts:
        name = part.partName or f"Part {part.id}"
        notes = part.recurse().getElementsByClass("Note")
        counts[name] = len(notes)
    return counts
```

## Open Questions

1. **Transposition verification strategy**
   - What we know: The big band preset has well-defined transpositions (Bb trumpet, Eb alto/bari sax, etc.). The MusicXML from the JSON assembler may or may not include `<transpose>` elements.
   - What's unclear: Whether the current `json_assembler.py` emits `<transpose>` elements in MusicXML output.
   - Recommendation: Inspect the JSON assembler output. If `<transpose>` is present, check it against expected values. If not, use key signature comparison as a proxy. If neither works reliably, downgrade check 6 to "manual verification recommended" in TESTPLAN.md. This is a v1 smoke test -- catching transposition errors is valuable but not blocking.

2. **Expected file counts for non-big-band presets**
   - What we know: Phase 7.1 only targets the big band preset (17 instruments). The expected ZIP file count is derived from this specific preset.
   - What's unclear: Should the harness support configurable expected counts for future presets?
   - Recommendation: Hard-code big band expectations for v1. Add a `--preset` flag later if needed. The harness can infer expected counts from the preset's instrument list, but building that abstraction now is premature.

3. **Web UI error recovery**
   - What we know: The pipeline can fail at any stage (normalization, separation, transcription, generation, rendering). The web UI needs to communicate failures to the user.
   - What's unclear: How much failure detail to show in the UI. The JSON smoke test output is designed for coding agents, not users.
   - Recommendation: Show a simple error message in the UI (e.g., "Pipeline failed during generation: compilation error after 5 fix attempts"). Log the full error to the server console. The user can share the error message with the developer for debugging.
