2. **Investigated vllm-mlx prefix caching** (source code analysis):
   - Prefix caching is NOT transparent — server computes `prefix_boundary` via LCP of two tokenizations
   - `engine/batched.py:641-688`: `_compute_prefix_boundary()` tokenizes messages twice (real + dummy with last user message replaced), finds longest common prefix
   - Cache is keyed by token sequences, saved at the boundary during two-phase prefill
   - **Client must structure messages so shared content comes BEFORE the last user message**
   - The `/v1/chat/completions` endpoint handles this automatically via `stream_chat()` → `_compute_prefix_boundary()`
   - LiteLLM sends standard chat completions, so prefix caching works transparently for chat API

3. **Identified the optimization needed** for Engrave prompts:
   - Current: system msg (800 tok shared) + ONE user msg (everything, varies per call)
   - Prefix boundary = only system message = ~800 tokens cached
   - Better: system msg (shared) + user msg 1 (per-song context, shared) + user msg 2 (per-section MIDI, varies)
   - This pushes boundary to ~1,000 tokens cached, only ~600 token suffix needs prefill
